## Findings considered and rejected

- **db.rs/storage.rs `ensure_table` duplication (DEBT-06)**: false positive —
  `db.rs:70-74` only delegates; `ensure_table` is called once, in
  `storage::insert`.
- **README status disclaimer (DOCS-01)**: README `## Status` (line 11) already
  states v0.1 is storage-layer-only; only micro-staleness remains (parser is
  now partially built). Not worth a plan.
- **Reserved system-table names (CORRECTNESS-3)**: no collision is physically
  possible — user table `t` maps to `_t_current`/`_t_history`, system tables
  have no such suffix. Revisit only if the physical naming scheme changes.
- **Identifier quoting defense-in-depth (SECURITY-1)**: injection is already
  blocked by the strict `validate_table_name` allowlist
  (`^[a-zA-Z_][a-zA-Z0-9_]*$`, `schema.rs:21`), applied at every entry point.
  Quoting all ~15 interpolation sites is churn without a current threat;
  revisit if validation rules ever loosen. (The cheap piece — clock cast
  bounds check — was folded into plan 003.)
- **Pagination / LIMIT (PERF-02)**: deliberate v0.1 scope; roadmap v0.2 adds
  LIMIT/OFFSET at the SQL layer, which is the right place for it.
- **`json(_data)` double-parse (PERF-05)**: `_data` is stored as JSONB, so
  `json()` is required to read it as text; eliminating the round-trip means
  changing the storage format. Unmeasured cost — benchmark before acting.
- **Split storage.rs god module (DEBT-01)**: ~half the 1710 lines are tests;
  premature to split before v0.2 grows the query layer. Plan 004's helper
  extraction is the incremental step.
- **SQL template registry (DEBT-03)**: `prepare_cached` (plan 004) removes the
  perf motivation; a registry adds indirection for little gain at this size.
- **IR too simple for roadmap (DEBT-04)**: the IR minimalism is recorded
  design (decision #21) with a planned `Filter::Where(Expr)` extension path;
  this is the roadmap, not debt.
- **Dependency bumps (DEPS-01)**: rusqlite 0.34 / sqlparser 0.55 are behind
  latest, but both are stable and nothing blocks on newer versions. Bump
  sqlparser opportunistically at the start of the next parser slice (after
  plan 005), when its test suite gives instant feedback on AST breakage.
- **Concurrency stress tests (TEST-03)**: worthwhile but below the cut line;
  the single `Mutex<Connection>` design makes races unlikely by construction.
  Reconsider when a reader-pool upgrade (decision #13) is attempted.
- **Parser fuzz target (direction)**: decision #3 explicitly isolates the
  parser so it "can be tested, fuzzed, and benchmarked independently." Add a
  `cargo-fuzz` target once the hand-rolled INSERT document-literal parser
  lands (that's where fuzzing pays; sqlparser-rs paths are already fuzzed
  upstream).
- **SQL dispatch/parser test gaps (TEST-01/TEST-02)**: subsumed by plan 005's
  test plan; each future v0.1b slice must ship its own end-to-end tests the
  same way.
